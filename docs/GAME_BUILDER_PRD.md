# Game Builder Layer — Product Requirements Document

**Version:** 1.0
**Status:** Source of Truth for Implementation
**Last Updated:** 2026-05-10
**Base project:** TUI Builder (this repository)

---

## 0. Reading guide

This PRD describes a **strictly additive** layer on top of the existing TUI Builder. Every section is grounded in the current codebase (file paths, state names, and patterns are real, not aspirational). Anything that conflicts with the actual code is a bug in this document — please report it.

The non-negotiable rule for implementation: **do not modify existing application behavior**. The Game Builder may *extend* registries, *add* new component files, *add* new state branches, and *conditionally render* new UI, but it must not rewrite or restructure existing modules.

---

## 1. Executive Summary

The **Game Builder Layer** extends TUI Builder with the ability to author 2D games (top-down, platformer, isometric, board, card, arcade) inside the same editor. It reuses the existing paradigms: drag-and-drop from a Toolbox onto a Canvas, property editing through the Inspector, screens managed via the User Journey Panel, persistence through the existing project JSON, and data through the existing in-memory database panel.

A toggle exposes a **Game Mode** that:
- Adds new game-specific components to the Toolbox (without removing existing ones).
- Adds a new **Level Tabs** strip above the Canvas when a game-typed screen ("World") is selected.
- Adds new Inspector branches for game-specific selections (Level, GameEntity, TileMap, etc.).
- Adds a **Sprite Sheet Manager** modal alongside Database / Projects / Settings.
- Extends the export pipeline to optionally produce a playable game.

Games can be embedded inside any existing TUI Builder Screen as a new `GameView` component.

## 2. Current state baseline (what exists today)

This section is the load-bearing reference for everything that follows. All identifiers below are real and were verified against the current code.

### 2.1 Tech stack
- React 18, Vite, plain CSS with CSS variables.
- `react-dnd` + `react-dnd-html5-backend` for drag and drop.
- `sql.js` available as a dependency, but the database panel today uses an in-memory JS object (not a real SQLite VM).
- `jszip` available for export bundling.
- No TypeScript, no test runner, no linter configured.

### 2.2 Top-level files
- `src/App.jsx` — main application (≈2,233 lines). Owns all top-level state, handlers, defaults, and the export pipeline (`exportHTML`).
- `src/components/Canvas.jsx` — canvas rendering and DnD targets.
- `src/components/Inspector.jsx` — properties panel, switches by selection type and component type.
- `src/components/Toolbox.jsx` — flat array of palette items.
- `src/components/DatabasePanel.jsx` — schema/data/query tabs over an in-memory store.
- `src/components/ProjectManager.jsx` — project list (localStorage).
- `src/components/IconPicker.jsx` — icon selector for `Image` component.
- `src/components/Componentes/*.jsx` — 23 component implementations.

### 2.3 State model (App.jsx)
Top-level React state actually present today:

| State | Shape | Notes |
|---|---|---|
| `screens` | `Array<Screen>` | Each screen has `{ id, name, rows, settings }`. |
| `currentScreenId` | `string` | UUID of active screen. |
| `selectedIds` | `string[]` | Multi-selection of component IDs. |
| `viewMode` | `'desktop' \| 'mobile'` | Canvas viewport. |
| `theme` | `'theme-nano' \| 'theme-bios' \| 'theme-retro' \| 'theme-amber'` | CSS-vars based. |
| `activeWindow` | `string \| null` | Visible Window when multiple exist. |
| `database` | `{ tables: [], data: {} }` | In-memory store. |
| `canvasPadding` | `{ top, right, bottom, left }` | |
| `clipboard`, `history`, `historyIndex` | — | For copy/paste and undo/redo. |
| `showUserJourney`, `showDatabase`, `showSettings`, `showProjects` | booleans | Modal/panel toggles. |

IDs are generated by `mkId()` (`crypto.randomUUID()` or fallback). Screen / row / component IDs all share this scheme.

### 2.4 Screen / Row / Component shape
```jsonc
// Screen
{
  "id": "uuid",
  "name": "Screen 1",
  "rows": [ /* Row */ ],
  "settings": { "timeout": 0, "nextScreenId": null, "webTitle": "", "metaTags": "" }
}

// Row (the layout container — never appears in Toolbox; rows are auto-created when dropping into the canvas)
{
  "id": "uuid",
  "layout": { "direction": "row|column", "gap": 8, "align": "...", "justify": "...", "wrap": false, "paddingTop": 0, "paddingRight": 0, "paddingBottom": 0, "paddingLeft": 0, "paddingLinked": true },
  "children": [ /* Component */ ]
}

// Component
{
  "id": "uuid",
  "type": "Window | Frame | Row | Button | Text | TextBox | CheckBox | RadioButton | ComboBox | ListBox | HScrollBar | VScrollBar | Timer | Shape | Line | Image | Table | Data | Loader | Tabs | Overlay | DataRepeater | Form",
  "props": { /* type-specific props, plus `sizing: { widthMode, heightMode }` and (for containers) `layout` */ },
  "children": [ /* nested components, only for CONTAINER_TYPES */ ]
}
```

Container types are declared in `Canvas.jsx`:
```js
const CONTAINER_TYPES = ['Window','Frame','Row','Tabs','DataRepeater','Form'];
```

The component implementation registry is also in `Canvas.jsx`:
```js
const componentMap = { Window, Frame, Row, Button, Text, Label: Text, Input: TextBox, TextBox, CheckBox, RadioButton, ComboBox, ListBox, Timer, Shape, Line, Image: ImageComp, HScrollBar: ScrollBar, VScrollBar: ScrollBar, Data, Table, Loader, Tabs, Overlay, DataRepeater, Form };
```

Default props per type live in `App.jsx` `getDefaultProps(type)` (around line 314).

### 2.5 Layout model
The editor uses **CSS Flexbox auto-layout exclusively**. There is **no absolute pixel positioning system today**. Every component has a `sizing.widthMode` / `sizing.heightMode` (`fixed | fill | hug`). This is critical for the Game Builder, because games typically need free 2D placement — see §6.5 (Coordinate Systems).

### 2.6 Drag-and-drop drag types
Defined across `Toolbox.jsx`, `Canvas.jsx`, and `App.jsx`:
- `'COMPONENT'` — drag from Toolbox (new component).
- `'EXISTING_COMPONENT'` — drag inside Canvas (move/reorder).
- `'SCREEN_CARD'` — drag in User Journey Panel (reorder screens).

### 2.7 Inspector
A single switch on selection type: nothing selected → Screen settings; a Row → Row layout panel; a Component → switch on `component.type` over ~25 cases. Adding a new component type requires:
1. A new `case '<Type>':` block in `Inspector.jsx` `renderProps()`.
2. A default-props entry in `App.jsx` `getDefaultProps`.
3. Adding the type to the registry in `Canvas.jsx` (`componentMap`, and `CONTAINER_TYPES` if applicable).
4. Adding a Toolbox entry in `Toolbox.jsx`.

### 2.8 User Journey Panel
Implemented inline in `App.jsx` as `UserJourneyPanel` (≈line 2202). Slide-out right sidebar, draggable screen cards, "+ Add Screen" tile, in-place rename, click to select, delete (cascade-confirmed and disallowed when only one screen remains). Toggled by the `≡ Journey` button (`.uj-toggle`).

### 2.9 Tabs component (relevant precedent)
`src/components/Componentes/Tabs.jsx` supports drag-rearranging *components inside the active tab*, but **its tab labels themselves are NOT drag-reorderable in the editor today**; the Inspector edits them as a comma-separated text field. Implication: the Level Tabs strip described in §5 is a **new UI element**, not a reuse of the `Tabs` component.

### 2.10 Toolbar / top area
The toolbar (`App.jsx` ≈ line 1786) contains: theme buttons, Projects button, Database button, Export button, view-mode select, save status. The `≡ Journey` button floats on the right edge. There is no dedicated "top-right corner" area today; the Game Mode toggle in §3 specifies exactly where it goes.

### 2.11 Database
`DatabasePanel.jsx` manages an object `{ tables: [...], data: {...} }` with three sub-tabs (Tables / Data / Query). `Form` reads from `sourceTable` and writes to `targetTable`. `DataRepeater` reads from `tableName` and renders its template once per record. `Table` can be database-bound via `dataSourceType: 'database'`.

### 2.12 Persistence
Auto-save with 1s debounce posts the full project JSON to `POST /api/projects` (see `App.jsx` `triggerSave`) and mirrors `currentProject` into `localStorage['nanostudio_current_project']`. Files in `projects/*.json` are server-side persisted snapshots.

### 2.13 Export
`exportHTML` in `App.jsx` walks each screen and emits a single self-contained HTML document with embedded CSS and JS. Only static interactivity is supported today (button actions: screen nav, overlay, email, external link, form submit; tab switching; basic data binding). There is **no game runtime** of any kind.

### 2.14 Keyboard shortcuts
Already wired in `App.jsx` keydown handler: `Delete`/`Backspace` (delete selection), `Cmd/Ctrl+Z` (undo), `Cmd/Ctrl+Shift+Z` / `Cmd/Ctrl+Y` (redo), `Cmd/Ctrl+D` (duplicate), `Cmd/Ctrl+C` (copy), `Cmd/Ctrl+V` (paste), `Cmd+Click` (navigate into Window).

### 2.15 Themes
Four CSS-variable themes (`THEMES` in `App.jsx`): Nano, BIOS, Retro, Amber. All component CSS uses `var(--bg)`, `var(--text)`, `var(--border)`, `var(--accent)`, etc.

## 3. Game Mode toggle

### 3.1 Placement
Add **one new toolbar button** in the existing `.toolbar` (`App.jsx` ≈ line 1786), positioned **after the theme buttons and before "Projects"**. Label: `Game Mode` (toggle state). When active, the button has the existing `.active` class, matching the visual treatment of theme buttons.

Rationale: there is no top-right "corner" today. Adding the toggle to the existing toolbar matches the established pattern (Database, Projects, Export are all toolbar buttons) and avoids inventing a new region.

### 3.2 State
Add to `App.jsx`:
```js
const [gameMode, setGameMode] = useState(false);
```
This flag is **per project** (saved into the project JSON at the top level: `gameMode: boolean`). Default `false` for backwards compatibility — every existing project in `projects/*.json` continues to load and behave exactly as before.

### 3.3 Behavior when `gameMode === true`
- Toolbox is *extended* with the game components defined in §4.2 (existing entries remain visible).
- The Level Tabs strip (§5) renders above the Canvas.
- The User Journey Panel labels switch from "Screen" → "World" cosmetically (§4.1); the underlying data shape does not change.
- The Sprite Sheet Manager toolbar button (§7) becomes visible. It is hidden in non-game projects to avoid noise.
- Inspector branches for new component/selection types become reachable.

### 3.4 Behavior when `gameMode === false`
Identical to current behavior. None of the additions above are visible. New game components are not in the Toolbox. Existing projects load with `gameMode: false` (defaulted on load if absent).

## 4. Worlds (reuse of existing Screens)

### 4.1 Worlds = Screens with `kind: 'world'`
Rather than introducing a parallel `worlds` array (which would fork persistence, duplicate the User Journey Panel, and break export), **a World is a Screen with a new optional field**:

```jsonc
{
  "id": "uuid",
  "name": "Forest",
  "kind": "world",                 // NEW. Optional. Defaults to "screen".
  "rows": [],                      // unchanged — used for HUD/menu overlays inside the world
  "levels": [ /* Level */ ],       // NEW. Only present when kind === "world".
  "currentLevelId": "uuid|null",   // NEW. Only present when kind === "world".
  "settings": { /* unchanged */ }
}
```

Backwards compatibility: an existing screen has no `kind` field → treat as `"screen"`. No migration needed.

### 4.2 User Journey Panel changes
**Behavioral change: none.** The same `UserJourneyPanel` component renders the same draggable cards with the same handlers. The only changes are cosmetic and conditional on `gameMode`:
- Header label: `[ USER JOURNEY ]` → `[ WORLDS ]`.
- "Add Screen" tile label → "Add World".
- New screens added while `gameMode` is on are created with `kind: 'world'` and an empty `levels: []`.

The same panel, when `gameMode` is off, continues to add `kind: 'screen'` (or omits the field). A single project may mix screens and worlds — useful when a game has menu screens plus gameplay worlds.

### 4.3 Inspector for a World
When a World is the selection (no component, no row), the Inspector shows the same Screen settings it shows today, plus a new **WORLD** section:

| Field | Default | Notes |
|---|---|---|
| World Name | inherits from screen.name | Reuses existing rename. |
| Theme music (sprite-sheet-manager Sound asset id) | `null` | See §7. |
| Default viewport type | `topdown` | `topdown \| platformer \| isometric \| board`. Inherited by new Levels unless overridden. |
| Default gravity | `0` | Numeric. Inherited by new Levels. |

## 5. Level Tabs

### 5.1 What and where
A horizontal tab strip rendered **between the toolbar and the Canvas** when (a) `gameMode === true` AND (b) the currently selected screen has `kind === 'world'`.

```
┌─ toolbar ──────────────────────────────────────────────────┐
│ [Themes] [Game Mode*] [Projects] [Database] [Sprites] ...  │
├─ level tabs (only when current screen.kind === 'world') ──┤
│ ▎ Level 1 │ Level 2 │ Level 3 │ + Add Level │              │
├─ canvas ───────────────────────────────────────────────────┤
│                                                            │
│   (renders activeLevel's TileMap + GameEntities, or         │
│    falls back to the world's `rows` when no level is        │
│    selected — used to author HUD/menu overlays.)            │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 5.2 Implementation
- New file: `src/components/LevelTabs.jsx`.
- Drag type: `'LEVEL_TAB'` (mirrors the precedent of `'SCREEN_CARD'`).
- Reuses the visual language of the User Journey screen cards. No new design tokens.
- "+ Add Level" button mirrors "+ Add Screen".
- Active tab highlighted via the existing `.active` modifier.
- Right-click (or Delete key while a tab is focused) → confirm and remove.
- Duplicate: `Cmd/Ctrl+D` while a tab is focused — reuses the existing duplicate plumbing in App.jsx by extending it to handle level selections.

### 5.3 Level data shape
```jsonc
{
  "id": "uuid",
  "name": "Level 1",
  "viewport": "topdown|platformer|isometric|board",
  "gravity": 0,
  "backgroundMusicAssetId": null,
  "spawnPointId": null,                 // a GameEntity id of role "spawn"
  "tileMap": {                          // inline; one TileMap per Level
    "tileWidth": 32,
    "tileHeight": 32,
    "cols": 30,
    "rows": 17,
    "tilesetAssetId": null,
    "layers": [
      {
        "id": "uuid",
        "name": "Background",
        "kind": "tiles | collision | spawn | overlay",
        "data": []                      // length cols*rows; tile indexes or 0
      }
    ]
  },
  "entities": [ /* GameEntity instances placed in this level */ ]
}
```

### 5.4 Level Inspector
When a Level tab is the active selection (new selection kind: `level`), the Inspector renders the fields above, plus the TileMap parameters (`tileWidth`, `tileHeight`, `cols`, `rows`, `tilesetAssetId`). All editing happens in the Inspector — no inline editing, matching the existing convention.

### 5.5 Selection model extension
Today selection is "screen | row | component". Add a fourth selection kind: `level`. Implement as:
```js
const [selectedLevelId, setSelectedLevelId] = useState(null);
```
Selecting a level clears `selectedIds` (mutually exclusive); selecting a component clears `selectedLevelId`.

## 6. New components (Toolbox additions in Game Mode)

All new components live under `src/components/Componentes/` to match the existing convention. Each requires the standard four-step registration (Toolbox entry, `componentMap`, default props, Inspector case).

### 6.1 Component list

| Component | File | Container? | Purpose |
|---|---|---|---|
| `GameView` | `Componentes/GameView.jsx` | No | Embeds a referenced World (by id) as a runnable game inside any Screen. The bridge from "app" to "game". |
| `TileMap` | `Componentes/TileMap.jsx` | No | Editor surface for painting tile layers. Lives at the Level root and is rendered automatically by Level view; not normally placed manually. |
| `GameEntity` | `Componentes/GameEntity.jsx` | No | Universal in-world entity. Role configured in Inspector (see §6.3). |
| `CollisionShape` | `Componentes/CollisionShape.jsx` | No | Box / circle / polygon collider attached to an entity or placed in world. |
| `Trigger` | `Componentes/Trigger.jsx` | No | Area + condition + actions (see §8). |
| `Teleporter` | `Componentes/Teleporter.jsx` | No | Trigger preset that targets `(worldId, levelId, spawnPointId)`. |
| `SpawnPoint` | `Componentes/SpawnPoint.jsx` | No | Marks an in-level position; referenced by Teleporters and Level defaults. |
| `Camera` | `Componentes/Camera.jsx` | No | Viewport behavior (follow target, bounds, zoom). At most one per Level. |
| `ParticleEmitter` | `Componentes/ParticleEmitter.jsx` | No | Visual particle source. |
| `SoundEmitter` | `Componentes/SoundEmitter.jsx` | No | Positional audio source. |

### 6.2 Toolbox grouping
The Toolbox today is a flat array. To preserve scannability when game components are added, introduce **section headers** (visual only, non-draggable). The shape becomes:
```js
const componentes = [
  { kind: 'section', label: 'UI' },
  { kind: 'item', type: 'Window', label: 'Window' },
  // ... existing items
  { kind: 'section', label: 'Game', requires: 'gameMode' },
  { kind: 'item', type: 'GameEntity', label: 'Game Entity', requires: 'gameMode' },
  // ...
];
```
The `requires: 'gameMode'` filter is applied at render time. Existing items get no `requires` and remain always visible. This preserves all current behavior.

### 6.3 GameEntity — universal entity
Following the established pattern (one type, role configured via prop), GameEntity has a single `type: 'GameEntity'` and a `role` prop that drives Inspector behavior. Default props:

```js
GameEntity: {
  role: 'prop', // 'playerMain' | 'player2' | 'player3' | 'player4' | 'enemy' | 'npc' | 'interactive' | 'prop'
  spriteSheetAssetId: null,
  defaultAnimation: 'idle',
  position: { x: 0, y: 0 },             // pixel coordinates within the Level
  facing: 'right',
  stats: { hp: 100, speed: 100, damage: 10 },
  persona: { /* see §6.4 */ },
  scriptId: null,                       // optional Lua script reference (§9)
  sizing: { widthMode: 'fixed', heightMode: 'fixed' }
}
```

The Inspector shows different sub-sections depending on `role`:
- `playerMain` / `player2..4`: input bindings, stats, persona.
- `enemy`: behavior preset (idle/patrol/chase/flee), persona.
- `npc`: dialog/quest hooks, persona.
- `interactive`: trigger on-collide, item drops.
- `prop`: visuals only.

### 6.4 GameEntity Persona schema
The Persona system is **metadata stored on the entity** that the runtime exposes to scripts and that future LLM-assisted generation can consume. It does not directly affect runtime behavior unless a script reads it. Schema (mirrors the original PRD §7 — kept here verbatim for completeness):

```jsonc
"persona": {
  "identity":   { "name": "", "species": "", "class": "", "functionalRole": "", "highConcept": "", "physicalSilhouette": "", "colorPalette": [] },
  "psychology": { "primaryMotivation": "", "moralAlignment": "", "coreValues": [], "fearsAndVulnerabilities": [], "dominantTrait": "" },
  "gameplay":   { "targetPlayerArchetype": "", "interactionLoop": "", "signatureMechanic": "", "powerCurve": "" },
  "narrative":  { "faction": "", "naturalAllies": [], "naturalEnemies": [], "worldImpact": "", "backstory": "" },
  "voiceAndPresence": { "voiceTone": "", "bodyLanguage": "", "signatureSound": "", "entryAnnouncement": "" }
}
```
The Inspector groups Persona under a collapsible **PERSONA** section to keep the panel scannable.

### 6.5 Coordinate systems (critical decision)
Existing components are positioned by Flexbox auto-layout. Game entities need **absolute pixel placement within a Level**.

**Decision:** game components opt out of the layout system. When the Canvas renders a Level (i.e., `currentLevel != null`), the canvas surface is positioned `relative` and entities render `absolute` at `position.x, position.y`. The existing `LayoutRow` machinery is **bypassed** for the Level surface. This keeps the two systems isolated and avoids leaking absolute positioning into application screens.

When a Level is *not* selected (i.e., the world's `rows` are being authored for HUD / menu overlay), the Canvas behaves exactly like today.

## 7. Sprite Sheet Manager (asset pipeline)

### 7.1 Why this is new
The project today has **no asset management** beyond the `Image` component's `src` field (URL or base64) and `IconPicker`. There is no concept of named, reusable assets. Game Builder needs one.

### 7.2 Scope
- New modal opened by a new toolbar button **Sprites** (visible only when `gameMode === true`).
- New file: `src/components/SpriteSheetManager.jsx`.
- New top-level state in `App.jsx`:
  ```js
  const [assets, setAssets] = useState({ sprites: [], sounds: [], tilesets: [] });
  ```
- New project JSON field: `assets`.

### 7.3 Asset shape
```jsonc
// SpriteSheet
{
  "id": "uuid",
  "kind": "spriteSheet",
  "name": "hero_walk",
  "src": "data:image/png;base64,...",
  "frame": { "width": 32, "height": 32, "cols": 8, "rows": 4 },
  "animations": [
    { "name": "idle", "frames": [0,1,2,1], "fps": 6, "loop": true, "hitbox": { "x": 4, "y": 6, "w": 24, "h": 26 } }
  ]
}
// Tileset, Sound — analogous.
```

### 7.4 Storage size warning
Base64 sprite sheets balloon project JSON quickly and will hit `localStorage` quotas (~5 MB). **Open question (§14):** decide whether assets are persisted inline in the project JSON, or extracted to a sidecar file (`projects/<id>.assets.json`), or moved to a real asset endpoint.

### 7.5 Editor surface
Per-asset:
- Upload PNG (or paste base64).
- Configure grid (frame width, height, cols, rows). Live preview overlays a grid on the image.
- Define animations: name, frame indexes, fps, loop flag, optional per-frame hitbox/pivot.
- Animation preview play/pause/scrub.

GameEntity `spriteSheetAssetId` references one of these by id.

## 8. Trigger and Teleporter system

### 8.1 Trigger
A `Trigger` component attached to a Level holds:
```jsonc
{
  "area": { "x": 0, "y": 0, "w": 64, "h": 64 },
  "conditions": [
    { "kind": "onEnter", "entityRoleFilter": "playerMain" }
    // also: "onCollide", "onItemCollected", "onFlag", "onTimer", "onEnemyDefeated"
  ],
  "actions": [
    { "kind": "showDialog", "text": "...", "duration": 2000 }
    // also: "teleport", "spawn", "playSound", "setFlag", "giveItem", "changeMusic", "cutscene", "runScript"
  ],
  "logic": "AND"  // all conditions must be true
}
```
Inspector lets you add/remove conditions and actions. Actions execute in declared order.

### 8.2 Teleporter
A `Trigger` preset — same data shape, but defaults to `condition: onEnter(playerMain)` and a single `teleport` action. Inspector exposes the target picker (World → Level → SpawnPoint). Optional `requires: { itemId | flagId }` gate.

### 8.3 Flags and inventory
Add to project JSON: `gameState.flags` (string→bool/number/string map) and `gameState.inventory` (array of `{ itemId, qty }`). These live in the runtime and are not user-edited at design time, except for setting initial values via the Inspector on the World.

## 9. Lua scripting (deferred to phase 2)

### 9.1 Storage (phase 1)
Scripts are **stored as strings** on Triggers (`runScript` action) and on GameEntities (`scriptId`). Editor side: a simple monospaced text area, no execution. This unblocks design-time authoring without committing to a runtime yet.

### 9.2 Execution (phase 2)
Bundle a Lua VM in the exported game runtime. Candidates: `wasmoon` (Lua 5.4 in WASM, ~250KB), `fengari` (Lua 5.3 in pure JS, ~140KB). Decision deferred (§14).

### 9.3 Exposed API (initial scope)
Same as the original PRD §12: `getPlayer`, `getDistanceTo`, `setBehavior`, `setAnimation`, `damage`, `playSound`, `dropItem`, `setFlag`, `getFlag`, `teleport`, `showDialog`. The API surface is the contract; the host implementation lives in the game runtime (§11).

---

## 10. Physics (deferred to phase 2 evaluation)

Lightweight integration. Three options to evaluate during phase-2 spike:
- **Custom AABB + tile collision** — minimal scope, fits the TileMap model, no dependency.
- **Planck.js** — ~150KB, deterministic, full Box2D feature set.
- **Matter.js** — ~80KB, easier API, less rigorous solver.

Initial recommendation: **custom AABB for top-down and platformer**, defer Planck/Matter unless a specific game requires constraints/joints. This keeps bundle size in line with the project's minimalist ethos.

Per-Level `gravity` field already covers the platformer use case. `viewport: 'topdown' | 'isometric' | 'board'` implies `gravity = 0`.

## 11. Game runtime

A new runtime module is required because the existing export pipeline emits static HTML/CSS/JS only.

- New file: `src/export/gameRuntime.js` (pure JS, framework-free; bundled into exports).
- Responsibilities: tick loop (requestAnimationFrame), entity updates, input, collision, camera, audio, animation, trigger evaluation, optional Lua dispatch.
- Editor preview: an in-editor "Play" button on the Level Tabs strip mounts the runtime in a `<canvas>` overlay so authors can test without exporting. Pressing Stop returns to the editor view.

This runtime is *also* what `GameView` (§6.1) embeds when used inside a TUI Builder Screen. Same code path for editor-preview, in-app embed, and standalone export.

## 12. Multiplayer (phase 3 — out of phase 1 scope)

### 12.1 Local multiplayer (phase 2)
Runtime supports up to four players by reading entities of role `player2..4` and binding configurable input slots (keyboard layout per player; gamepad slot if available). Optional split-screen viewport — a Camera component per player.

### 12.2 Network multiplayer (phase 3)
Out of scope for v1. Captured here so the Persona / GameEntity schema doesn't need re-design later. Decision between WebRTC peer-to-peer and WebSocket client-server is deferred.

## 13. Database integration

The existing `DatabasePanel` already supports the data Game Builder needs. To make the game/runtime contract explicit, **auto-create three tables** the first time `gameMode` is enabled in a project (only if they don't already exist):

| Table | Purpose | Default fields |
|---|---|---|
| `game_items` | Item catalog | `id INTEGER PRIMARY`, `name TEXT`, `description TEXT`, `iconAssetId TEXT` |
| `game_dialog` | Dialog lines (i18n-ready later) | `id INTEGER PRIMARY`, `key TEXT`, `text TEXT` |
| `game_progress` | Save data slots | `id INTEGER PRIMARY`, `slot INTEGER`, `flags TEXT`, `inventory TEXT`, `lastWorld TEXT`, `lastLevel TEXT` |

Auto-creation is **non-destructive** (skipped if tables exist) and **opt-in** (a one-time prompt the first time Game Mode is enabled). Existing tables and data are never modified.

## 14. Export

Two export modes (toggle on the existing Export button when `gameMode === true`):

| Mode | Output | Used for |
|---|---|---|
| **App Export** (existing) | Single self-contained HTML | Unchanged. Still works for non-game projects. |

## 17. Phased implementation plan

| Phase | Scope | Exit criteria |
|---|---|---|
| **0. Foundations** | `gameMode` flag, project JSON additions, toolbar toggle, no-op when off. | Existing projects load and behave identically. Toggle persists. |
| **1. Worlds + Levels (authoring only)** | World `kind`, Level data shape, `LevelTabs` component, Level Inspector, selection-kind extension. | Author can add a world, add levels, name them, reorder via DnD, switch between them. No runtime yet. |
| **2. Sprite Sheet Manager** | Asset state, manager modal, sprite/tileset/sound import, animation editor, asset references. | Author can import a sprite sheet, define animations, preview them. |
| **3. Game components — placement** | `TileMap`, `GameEntity`, `SpawnPoint`, `Camera`, `Trigger`, `Teleporter`, `CollisionShape`. Absolute-positioning canvas mode. | Author can paint tiles, place entities, configure triggers/teleporters in Inspector. No runtime yet. |
| **4. Game runtime + playtest** | `gameRuntime.js`, in-editor Play button, top-down + platformer viewports, basic AABB collision, animation playback, audio, trigger evaluation. | Author presses Play, controls the player, walks between levels via teleporters, hears sounds. |
| **5. Embedding + export** | `GameView` component (embeds a world into a Screen), Game Export mode with zip bundle. | Exported standalone game runs in a browser. Embedded GameView plays inside an exported app. |
| **6. Lua scripting** | Bundle Lua VM, expose API surface, wire `Trigger.runScript` and entity `scriptId`. | A script attached to an enemy makes it patrol; a trigger script sets a flag and changes music. |
| **7. Local multiplayer** | Multi-player input bindings, optional split-screen via multiple Cameras. | Two players control distinct entities on the same machine. |
| **8. Particles + polish** | `ParticleEmitter`, `SoundEmitter` positional audio, transition effects on teleport. | Visual feedback complete. |
| **(deferred)** | Network multiplayer, advanced physics, board/card-game presets, isometric viewport. | Not in v1. |

## 19. Risks and open questions

| # | Question | Why it matters | Owner / when to decide |
|---|---|---|---|
| Q1 | **Asset persistence model** — inline base64 in project JSON, sidecar JSON file, or asset endpoint? | Project JSON balloons; localStorage quota; save speed. | Phase 2 kickoff. |
| Q2 | **Lua VM choice** — wasmoon vs fengari vs no-Lua-only-JS? | Bundle size and DX. | Phase 6 spike. |
| Q3 | **Physics library** — custom vs Planck vs Matter? | Bundle size, determinism, scope. | Phase 4 spike. |
| Q4 | **Coordinate-system isolation** — confirm absolute positioning lives only inside Level rendering and never leaks into Screen authoring. | Architectural integrity. | During Phase 3 implementation. |
| Q5 | **Editor performance** — TileMap with thousands of tiles in React DOM will be slow. Likely need a `<canvas>` for the TileMap surface. | UX. | Phase 3 spike. |
| Q6 | **Undo/redo for tile painting** — current history granularity is per-state-change; tile painting is high-frequency. Coalesce strokes? | UX, memory. | Phase 3. |
| Q7 | **Embedded game lifecycle** — when a Screen with a `GameView` is exited, does the game pause, reset, or persist state? | Player expectations. | Phase 5. |
| Q8 | **Game Mode persistence scope** — per project (chosen here) vs per editor session vs global setting? | Project portability. | Locked in this PRD as **per project**. |
| Q9 | **Auto-created tables (game_items / game_dialog / game_progress)** — opt-in prompt vs silent. | Surprising side effects. | Locked here as **opt-in prompt**. |

## 21. Glossary

| Term | Definition |
|---|---|
| **Screen** | The pre-existing top-level page concept. Unchanged. |
| **World** | A Screen with `kind: 'world'` and a `levels` array. |
| **Level** | A unit of gameplay inside a World. Has a TileMap and entities. |
| **GameEntity** | Universal in-world object (player/enemy/npc/interactive/prop). Role decides Inspector branch. |
| **Persona** | Metadata on a GameEntity describing identity, psychology, gameplay, narrative, voice. |
| **Asset** | A reusable named resource (sprite sheet, tileset, sound) in the new asset registry. |
| **Game Mode** | A boolean per project that enables the Game Builder UI. |
| **Game Runtime** | The framework-free JS module that executes a game in the editor preview, in `GameView`, and in standalone exports. |
